# COMPONENT STYLING

# Pit Wall Insight

Version: 1.0

Status: Living Document

---

# Purpose

This document defines the appearance and behavior of every UI component used throughout Pit Wall Insight.

Every component should feel like part of one complete engineering system.

Users should never feel as though different components were designed by different people.

Consistency is a feature.

---

# Component Philosophy

Components are tools.

Not decoration.

Every component exists to

• Present information

• Enable interaction

• Reduce cognitive load

Nothing else.

# Visual Language

# Component Hierarchy

Primary

↓

Secondary

↓

↓

---

# Border Philosophy

Borders define structure.

Not decoration.

Borders should remain

Thin

Sharp

Low Contrast

Consistent

Hover may increase contrast.

Never thickness.

---

# Radius

Rounded corners should remain subtle.

No exaggerated rounding.

The interface should feel engineered.

Not soft.

---

# Shadows

Shadows communicate elevation.

Not style.

Use

Soft

Minimal

Controlled

Never dramatic.

Never floating.

---

# Buttons

Buttons represent actions.

Nothing more.

---

## Primary Button

Reserved for

Primary actions.

One primary button per section whenever possible.

Characteristics

Constructor Accent

High Contrast

Minimal Shadow

Sharp Focus

---

## Secondary Button

Neutral appearance.

Supports the primary action.

Should never compete visually.

---

## Ghost Button

No background.

Border only when necessary.

Hover introduces subtle emphasis.

---

## Icon Button

Minimal.

Square.

Centered.

Perfect alignment.

Icons remain visually balanced.

---

Collapsible.

Wide

# Inputs

Inputs should feel

Precise

Technical

Clean

not animation.

# Dropdowns

Dropdowns should resemble engineering controls.

Minimal transitions.

Clear hierarchy.

No oversized menus.

---

# Selectors

Circuit

Season

Session

# Tabs

Tabs divide context.

Not content.

Only one active tab should exist.

Active state uses constructor accent.

---

# Tables

Tables should feel like professional telemetry software.

Minimal borders.

Excellent alignment.

Readable spacing.

Never striped.

Never colorful.

---

# Table Rows

↓

↓

---

# Pagination

Simple.

Minimal.

Fast.

Never dominate layouts.

---

# Tooltips

Tooltips answer

"What am I looking at?"

Nothing more.

Keep concise.

---

# Progress Indicators

Minimal.

Linear.

Engineering inspired.

Avoid circular progress unless context requires it.

---

# Progress Bars

Thin.

Accurate.

Constructor accent.

Never gradients.

---

# Dialogs

Dialogs interrupt workflows.

Use only when necessary.

Background fades.

Focus shifts.

Interaction becomes clear.

---

# Drawers

Reserved for supporting information.

Do not replace pages with drawers.

---

# Accordions

Only when information density requires them.

Animations remain subtle.

---

# Notifications

Quiet.

Minimal.

Never intrusive.

Users should acknowledge,

not fear,

notifications.

---

# Toast Messages

Bottom or top corner.

Disappear naturally.

Never block workflows.

---

# Empty States

Minimal illustration.

Clear explanation.

Primary action.

Avoid humor.

---

# Loading Components

Skeletons resemble the final layout.

Avoid generic placeholders.

---

# Dividers

Used sparingly.

Whitespace is preferred.

---

# Icons

Thin stroke.

Consistent size.

Mechanical appearance.

No filled icons unless required.

---

# Images

Always purposeful.

Photography should feel premium.

Never stock-photo-like.

# Do

✓ Maintain consistency.

✓ Prioritize readability.

✓ Keep interactions predictable.

✓ Let typography lead.

✓ Use constructor colors sparingly.

✓ Respect spacing.

---

# Don't

✗ Decorative components.

✗ Excessive shadows.

✗ Bright gradients.

✗ Floating glass effects.

✗ Multiple border styles.

✗ Random corner radii.

✗ Different interaction patterns.

---

# Final Principle

Every component should feel as though it was engineered for one purpose—

to help users understand Formula One performance with confidence, precision, and speed.

If a component draws more attention to itself than to the information it contains,

it should be redesigned.
